## 3. 当前已知未解决问题 / 未完成能力

以下内容不是“代码跑不起来”，而是当前版本尚未补齐的工程能力或产品能力。

### 3.1 没有鉴权

当前所有 HTTP 管理接口和 MCP tools 都没有鉴权控制。

这意味着理论上任何能访问服务的人都可以：

- 创建 service
- 发送飞书消息
- 导入知识库
- 调用大模型
- 创建定时任务

这对于本地验证不是问题，但对于公网环境是高风险点。

### 3.2 没有飞书事件去重

当前没有对飞书事件做 `message_id` 级别去重。

可能产生的问题：

- 飞书重试导致重复回复
- 重复抓图
- 重复写入知识库
- 重复群聊总结

### 3.3 不能自动枚举“机器人所在所有群”

当前支持：

- 指定 `chat_id` 抓群聊
- 当前群快捷操作

但还不支持：

- 自动列出机器人加入的所有群
- 一键抓取所有群消息

### 3.4 不是向量知识库

当前知识库实现是：

- SQLite
- FTS5 全文检索

而不是：

- embedding
- 向量召回
- rerank

所以在语义召回上会弱于真正的向量检索方案。

### 3.5 没有 OCR / 结构化抽取

当前图片能力主要是：

- 图片抓取
- 视觉模型自然语言分析
- 分析结果写回知识库

但还没有：

- OCR 抽取
- 结构化字段抽取
- 风险项 / 待办项自动结构化

### 3.6 没有异步任务队列

当前长操作主要靠：

- FastAPI 背景任务
- MCP 内部 scheduler

还没有：

- Celery
- RQ
- Kafka / RabbitMQ 异步队列

### 3.7 缺少系统级可观测性

当前没有系统化接入：

- metrics
- tracing
- 结构化日志
- 报警

本地开发和小规模验证够用，但线上排障会比较吃力。

## 4. 高风险点（部署和联调时最容易出问题）

以下风险是当前项目里最值得优先注意的。

### 4.1 没有公网 HTTPS，飞书回调一定不通

飞书要求回调地址能公网访问，且通常需要 HTTPS。

如果你只是本地起服务：

- `http://127.0.0.1:8000`

飞书是打不进来的。

所以如果你要验证飞书消息回调，必须满足下面任一条件：

- 用 `cloudflared tunnel`
- 用 `ngrok`
- 用自己的公网机器 / 域名 / HTTPS 反向代理

### 4.2 文本模型能用，不代表视觉模型一定能用

当前图像分析走的是 OpenAI 兼容的多模态消息格式：

- `messages[].content = [{type:text}, {type:image_url}]`

不是所有 OpenAI 兼容网关都支持这套 vision 输入格式。

可能出现：

- 文本问答成功
- 图像分析失败

所以本地验证时必须确认：

- 你接的后端模型服务真的支持 vision

### 4.3 MCP 起了，不代表定时任务一定马上成功

`docker-compose.yml` 里 MCP 依赖主服务：

- 有 `depends_on`

但没有基于健康检查的等待逻辑。

因此首次启动时，可能出现：

- MCP scheduler 先起来
- 主服务还没 fully ready
- 第一轮定时任务调用失败一次

### 4.4 明文存密钥

当前数据库中明文保存：

- 飞书 `app_secret`
- 模型 `llm_api_key`

本地验证可接受，但生产环境有明显安全风险。

### 4.5 无鉴权直接公网暴露风险高

如果你直接把当前版本暴露到公网，任何能访问服务的人理论上都能：

- 发消息
- 建 service
- 跑定时任务
- 调模型

所以**本地测试可以直接用，公网环境必须加鉴权**。

### 4.6 飞书回调 accepted 但机器人不回复，容易误判为“项目坏了”

当前回调处理逻辑是：

- HTTP 先快速返回 `accepted`
- 再在后台异步处理

如果后台处理异常：

- 前台 HTTP 看起来是成功的
- 但机器人不会回消息

你需要看应用日志才能定位原因。

## 5. 你本地部署最可能踩的坑

### 坑 1：服务健康检查通过，但飞书完全没反应

最常见原因：

- 你没有公网 HTTPS
- 回调 URL 不能被飞书访问
- `service_id` 配错
- 飞书后台 token / encrypt key 跟 service 配置不一致

### 坑 2：文本问答 OK，图片分析失败

最常见原因：

- 模型不支持 vision
- 网关不支持 `image_url` 格式
- 图片太大
- base64 数据不符合后端要求

### 坑 3：MCP 能起，但定时任务没看到效果

最常见原因：

- 你用的是 `stdio` 模式，MCP 不是常驻
- MCP 进程没持续运行
- 主服务地址没配对
- 首轮任务执行时主服务还没就绪

### 坑 4：图片分析或群聊总结能返回结果，但后续检索不到

最常见原因：

- 没打开 `save_analysis_to_knowledge_base`
- 没打开 `save_summary_to_knowledge_base`
- 检索词与写入内容差距太大，FTS 没命中

## 6. 推荐验证顺序

为了避免定位困难，建议按下面顺序验证。

### 第一步：先只验证主服务，不接飞书

先起主服务：

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

检查：

- `/health`
- `/docs`

### 第二步：先验证 OpenAPI

建议在 `/docs` 里按顺序手测：

1. `POST /api/v1/services`
2. `POST /api/v1/services/{service_id}/llm/ask`
3. `POST /api/v1/services/{service_id}/llm/image-analyze`
4. `POST /api/v1/services/{service_id}/llm/image-analyze/upload`
5. `POST /api/v1/services/{service_id}/feishu/chats/summarize`

### 第三步：验证知识库

先导入一段文本，然后验证：

1. `POST /knowledge-base/text`
2. `GET /knowledge-base/search`
3. `POST /llm/ask` with `use_knowledge_base=true`

### 第四步：验证 MCP

先起主服务，再起 MCP：

```bash
python3 -m app.mcp_server
```

优先测试：

- `service_health`
- `ask_llm_question`
- `analyze_image_with_llm`

如果这些都通，再测：

- 定时任务
- 群聊总结

### 第五步：最后再接飞书

当主服务、模型、知识库、MCP 都通后，再做飞书联调：

1. 暴露公网 HTTPS
2. 创建 service
3. 配置飞书 callback
4. 配置 token / encrypt key
5. 订阅事件
6. 把机器人加入群
7. 测：
   - `帮助`
   - `抓取当前群`
   - `总结当前群`
   - 发图片

## 9. 总结判断

### 当前项目结构是否清晰？

是，结构清晰，主干已经完整。

### 当前项目是否已经具备验证条件？

- 本地 OpenAPI 验证

- 公网 HTTPS 后做飞书联调

### 现在最不适合直接做什么？

不建议当前版本直接裸奔公网正式使用，因为还缺：

- 鉴权
- 去重
- Secret 安全管理
